# Estándares de Código - Raioz

Este documento define las reglas y estándares de código que deben seguirse en el proyecto Raioz.

## 📏 Reglas Generales

### Límite de Líneas por Archivo

- **Máximo: 400 líneas por archivo**
- Si un archivo excede este límite, debe dividirse en múltiples archivos más pequeños
- Cada archivo debe cumplir un objetivo específico y único
- Excepción: Archivos generados automáticamente (marcados con comentario `// Code generated by...`)

### Longitud de Línea

- **Máximo: 120 caracteres por línea**
- Incluye espacios, indentación y código
- Si una línea excede este límite, debe dividirse de manera legible
- Priorizar legibilidad sobre cumplimiento estricto si dividir afecta la comprensión

### Objetivo Único por Archivo

- Cada archivo debe tener un propósito claro y único
- Si un archivo maneja múltiples responsabilidades, dividir en múltiples archivos
- Nombre de archivo debe reflejar su propósito

## 🐹 Estándares de Go

### Naming Conventions

#### Paquetes

- Nombres en minúsculas, una sola palabra
- No usar guiones bajos ni guiones
- Nombres cortos pero descriptivos
- Ejemplos: `config`, `docker`, `env`, `git`

#### Variables y Funciones

- Variables locales: camelCase
- Variables exportadas: PascalCase
- Funciones exportadas: PascalCase
- Funciones no exportadas: camelCase
- Acrónimos: todas mayúsculas o todas minúsculas según contexto (ej: `HTTP`, `API`, `ID`)

#### Constantes

- Constantes exportadas: PascalCase
- Constantes no exportadas: camelCase
- Constantes relacionadas: usar prefijo común o agrupar en bloque

#### Interfaces

- Interfaces exportadas: terminan en `-er` cuando es posible (ej: `Reader`, `Writer`)
- Nombres descriptivos de lo que hacen, no de cómo lo hacen
- Una sola responsabilidad por interfaz

#### Errores

- Variables de error: `err` o `Err` (si exportada)
- Tipos de error: terminan en `Error`
- Mensajes de error: minúsculas, sin puntuación final
- Usar `fmt.Errorf` con `%w` para wrapping

### Estructura de Código

#### Organización de Archivos

```
internal/
  package/
    types.go          # Definiciones de tipos
    functions.go      # Funciones públicas
    helpers.go        # Funciones privadas/helpers
    package_test.go   # Tests
```

#### Imports

- Orden: stdlib, terceros, locales (raioz)
- Agrupar con línea en blanco entre grupos
- Usar `goimports` para formateo automático

#### Comentarios

- Paquetes: comentario antes de `package` explicando propósito
- Funciones exportadas: comentario que empieza con nombre de función
- Funciones complejas: explicar lógica no obvia
- Comentarios en código: explicar "por qué", no "qué"

#### Funciones

- Máximo 100 líneas por función
- Máximo 50 statements por función
- Máxima complejidad ciclomática: 15
- Una responsabilidad por función
- Nombres descriptivos (verbos para funciones, sustantivos para getters)

#### Manejo de Errores

- Siempre verificar errores, no ignorarlos
- Usar `fmt.Errorf` con `%w` para wrapping
- Devolver error en último valor de retorno
- No usar `panic` excepto en casos extremos (inicialización)

#### Variables

- Declarar variables cerca de su uso
- Preferir `:=` cuando el tipo es obvio
- Inicializar variables si es necesario
- Evitar variables globales

### Buenas Prácticas

#### Performance

- Evitar asignaciones innecesarias en loops
- Usar `strings.Builder` para concatenación de strings
- Preferir `make()` con capacidad conocida para slices/maps
- Usar `defer` apropiadamente pero no en loops críticos

#### Concurrencia

- Usar `context.Context` para cancelación
- Cerrar recursos con `defer`
- Evitar goroutines sin control de finalización
- Usar channels apropiadamente

#### Testing

- Archivos de test: `*_test.go`
- Funciones de test: `TestXxx`
- Funciones de benchmark: `BenchmarkXxx`
- Usar tabla de tests cuando sea apropiado
- Tests deben ser determinísticos y aislados

#### Dependencias

- Minimizar dependencias externas
- Usar `go mod` para gestión de dependencias
- Versionar dependencias explícitamente
- Revisar dependencias regularmente

### Estructura de Directorios

#### Organización del Proyecto

```
raioz/
├── cmd/              # Comandos CLI
├── internal/         # Código interno (no exportable)
│   ├── config/       # Configuración
│   ├── docker/       # Integración Docker
│   ├── env/          # Variables de entorno
│   ├── git/          # Integración Git

│   ├── validate/     # Validación

```

### Herramientas y Linters

#### Herramientas Requeridas

- `golangci-lint`: Linter principal
- `goimports`: Formateo de imports
- `gofmt`: Formateo de código
- `go vet`: Detección de errores comunes

#### Comandos Útiles

```bash
# Formatear código
go fmt ./...

# Formatear imports
goimports -w .

# Ejecutar linter
golangci-lint run

# Ejecutar tests
go test ./...

# Build
go build -o raioz .
```

## 🔍 Reglas Específicas del Proyecto

### Manejo de Archivos

- Usar `path/filepath` para rutas, no `path`
- Validar existencia de archivos antes de leerlos
- Cerrar archivos con `defer`
- Manejar errores de I/O apropiadamente

### Logging

- Usar `fmt` para output de usuario
- Logs estructurados cuando sea necesario
- Niveles apropiados de logging

### Configuración

- Validar configuración temprano
- Proporcionar valores por defecto razonables
- Error claro cuando falta configuración requerida

### Integración Externa

- Timeouts en todas las operaciones de red
- Manejo apropiado de errores de comandos externos
- Validación de outputs de comandos externos

## 📝 Ejemplos

### ✅ Buen Código

```go
// Package config provides configuration loading and validation.
package config

import (
    "encoding/json"
    "fmt"
    "os"
)

// LoadDeps loads dependencies from a JSON file.
func LoadDeps(path string) (*Deps, error) {
    data, err := os.ReadFile(path)
    if err != nil {
        return nil, fmt.Errorf("failed to read config file: %w", err)
    }

    var deps Deps
    if err := json.Unmarshal(data, &deps); err != nil {
        return nil, fmt.Errorf("failed to parse config: %w", err)
    }

    return &deps, nil
}
```

### ❌ Mal Código

```go
package config

import "encoding/json"
import "os"
import "fmt"

func load(path string) (interface{}, error) {
    data, _ := os.ReadFile(path) // Error ignorado
    var x interface{}
    json.Unmarshal(data, &x)
    return &x, nil
}
```

- [ ] Nombres son descriptivos
